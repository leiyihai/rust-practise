# 用类型系统建模游戏状态

## 枚举就是状态机

文字冒险游戏中，游戏的各种状态可以直接编码为枚举：

```rust
enum GameState {
    Exploring,           // 自由探索
    Fighting(Enemy),     // 战斗中（携带敌人数据）
    Shopping(Shop),      // 购物中
    Dialoging(NpcId),    // 对话中
    GameOver(bool),      // 游戏结束（是否胜利）
}
```

`match` 就是一个**穷尽性的状态处理器**——编译器确保你不会漏掉"购物中忘记处理购买逻辑"这类 bug。

## 为什么不用继承

面向对象语言通常会这样建模：

```text
         GameState (抽象类)
        /    |     \
   Exploring Fighting Shopping  (子类)
```

但在 Rust 中，枚举 + match 的组合更强大：
- 不需要向下转型（downcast）
- 状态转换显式可控
- 可以携带不同数据（Exploring 无数据，Fighting 携带 Enemy）

## 类型状态模式（Type State Pattern）

更高级的做法：用**类型参数**编码状态，让非法操作在编译期被拒绝：

```rust
struct Player<State> {
    name: String,
    hp: u32,
    _state: PhantomData<State>,
}

struct Alive;
struct Dead;

impl Player<Alive> {
    fn take_damage(self, dmg: u32) -> Player<Alive> { ... }
    fn die(self) -> Player<Dead> { ... }
}

impl Player<Dead> {
    // 死人不能攻击——这个方法根本不存在
}
```

这样，`player.attack()` 如果 player 是 `Player<Dead>` 类型就**编译不过**——在用户写出 bug 之前就阻止了它。

## 触类旁通：ECS 架构

大型游戏（如 Rust 写的 Bevy 引擎）通常用 ECS（Entity Component System）而非继承：

```
Entity (一个 ID)
  ├── Component: Position { x, y }
  ├── Component: Health { current, max }
  └── Component: Name("哥布林")

System: fn damage_system(query: Query<&mut Health>) { ... }
```

Rust 特别适合 ECS，因为：
- Component 是纯数据（struct），天然适合所有权模型
- System 是纯函数，无副作用
- 借用检查确保 System 之间没有数据竞争
- 不需要虚函数/动态分发，性能极佳

如果你未来想用 Rust 做游戏，Bevy 的 ECS 范式值得了解。
