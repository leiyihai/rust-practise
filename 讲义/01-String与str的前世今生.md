# String 与 &str 的前世今生

## 它们到底是什么？

很多初学者以为 `String` 和 `&str` 是"同一个东西的两种写法"——完全错误。

`String` 是一个**拥有堆内存的结构体**，长得像这样：

```
String {
    ptr: 指向堆上 UTF-8 字节的指针,
    len: 当前字节数,
    cap: 已分配的容量,
}
```

`&str` 是一个**胖指针**（fat pointer），它不拥有数据：

```
&str {
    ptr: 指向某处 UTF-8 字节的指针,
    len: 字节数,
}
```

关键区别：`String` 负责在 drop 时释放堆内存；`&str` 只是借用别人的数据，不管释放。

## 内存长什么样

```text
栈                          堆
┌──────────────┐           ┌─────────────────┐
│ String       │           │ 'H' 'e' 'l' 'l' │
│  ptr ──────────────────> │ 'o' ' ' '世'    │
│  len = 11    │           │ '界'             │
│  cap = 16    │           └─────────────────┘
└──────────────┘
                    ┌──────────────┐
&str 指向字面量:      │ ptr ────────│──> 二进制里的 "hello"
                    │ len = 5     │
                    └──────────────┘
```

## 为什么 &str 不是 &String

`&String` 可以被自动解引用为 `&str`（通过 `Deref` trait），但它们是不同类型。函数参数优先用 `&str` 而非 `&String`，因为字符串字面量、切片、`String` 都能转为 `&str`，调用方更灵活。

## UTF-8 的坑

Rust 的字符串是 UTF-8 编码，所以：
- `"你好".len()` 返回 6（每个中文字符 3 字节），不是 2
- `"你好".chars().count()` 返回 2（字符数）
- 不能直接用索引 `s[0]`，因为一个字符可能占多字节
- 可以用 `s.chars().nth(0)` 取第 0 个字符

## String 的常用操作一览

| 操作 | 方法 | 是否分配新内存 |
|------|------|:---:|
| 拼接 | `push_str` / `push` | 可能扩容 |
| 格式化 | `format!` | 是（新 String） |
| 替换 | `.replace()` | 是 |
| 转切片 | `&s[..]` / `s.as_str()` | 否 |
| 转字节 | `s.into_bytes()` | 否（消费原值） |

## 触类旁通：其他语言的字符串

- **Go**: `string` 是不可变的字节序列，类似 `&str`；`strings.Builder` 对应 `String`
- **Python**: `str` 不可变，自动管理内存；没有借用概念
- **C++**: `std::string` 可变，类似 `String`；`std::string_view` 类似 `&str`

Rust 独特之处在于**编译期强制区分"拥有"和"借用"**——你不必记得谁负责释放内存，编译器替你记。
